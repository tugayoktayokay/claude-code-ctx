# v0.7 â€” Phase 2 measurement, cache GC, expanded Bash coverage

**Date:** 2026-04-21
**Target version:** 0.7.0
**Scope:** Aggregator + CLI for measuring whether `pre_tool_use` redirects actually work; cache GC for `mcp_cache`; expanded Bash rule set. No new MCP tools. No breaking changes to public CLI or config surface.

## Context

v0.6.0 shipped 15 `pre_tool_use` rules (4 `deny`, 11 `ask`) that steer heavy Bash calls toward `ctx_shell` / `ctx_read` / `ctx_grep`. The commit message explicitly scheduled Phase 2 to measure whether Claude obeys those redirects, but no measurement code landed.

Without measurement, every future tuning decision â€” new rules, threshold changes, deny/ask reclassification â€” is a guess. Adaptive thresholds, learning-from-session, and a proactive budget advisor (the "smart MCP" direction) all depend on the same correlation data as their input. Phase 2 is therefore the prerequisite for everything after.

Two secondary problems found during v0.6.0 review also fit the same release:

1. `mcp_cache` has no GC. Heavy sessions accumulate cache files under `~/.config/ctx/cache/` without bound.
2. The Bash regex set misses common variants (`rg`, `grep -R`, `egrep`, `awk/sed` over files, `wc -l` on globs, `find` without `-maxdepth`). Easy to route around today.

Native `Read` / `Grep` tool interception was considered and rejected: both are internally bounded (Read: 2000-line default; Grep: `head_limit: 250`). The real footguns live in Bash. Adding hooks for native tools is deferred to v1.0 if Phase 2 data shows it's needed.

## Decision

1. Extend `hooks.log` with `session=<id>` on `pre_tool` events and a new `post_tool` event per tool invocation.
2. Add a pure aggregator module `src/metrics.js` that reads `hooks.log`, correlates pre/post events by session + 60s window, and returns classified counts.
3. Add a `ctx metrics` CLI subcommand that prints the aggregated report (default range: last 7 days).
4. Add TTL + max-bytes + LRU GC to `src/mcp_cache.js`, tunable via `config.cache.gc`.
5. Add 6 new Bash rules to `config.default.json` covering the identified gaps.

Zero runtime dependencies. No LLM calls. No hook handler signature changes. No change to public MCP tool surface.

### Log schema

**pre_tool event** (existing format + new `session=` field):

```
<ISO8601> pre_tool session=<id> action=<deny|ask> tool=<name> pattern="<regex>" cmd_head="<first-80>" reason="<...>"
```

Missing session_id (old hook versions, edge cases) logs `session=-`.

**post_tool event** (new, same shape):

```
<ISO8601> post_tool session=<id> tool=<name> cmd_head_or_args_head="<first-80>" exit=<code> size_bytes=<int>
```

- `cmd_head` for `Bash` tool (first 80 chars of `tool_input.command`).
- `args_head` for non-Bash tools (first 80 chars of `JSON.stringify(tool_input)`).
- `size_bytes`: if `tool_response.content` is a string, its `.length`; else `JSON.stringify(tool_response).length`; `0` if `tool_response` is null/undefined.
- `exit` comes from `tool_response.exit_code` when available; `0` default when Claude Code did not supply one. For non-Bash tools `exit` is a sentinel and is ignored during correlation â€” the classification only reads it on `tool == "Bash"` matches.

**cache events** (new, emitted from `mcp_cache.js`):

```
<ISO8601> cache-write ref=<ref> bytes=<n>
<ISO8601> cache-read  ref=<ref> result=<hit|miss> bytes=<n>
<ISO8601> cache-gc    swept=<N> bytes_freed=<M>
```

Cache events do not include `session=` because the MCP tool handler signature does not receive Claude Code's session_id. These events are aggregated independently for the "cache" block of the report and are not correlated with pre/post events.

All event types share the single `~/.config/ctx/hooks.log` file. No separate log and no separate JSON stats file; every counter derives from the same log, so date-range filters apply uniformly.

### Correlation algorithm (`metrics.js`)

Input: path to `hooks.log`, range (default last 7 days), correlation window (default 60 seconds).

1. Forward-scan the log file line by line. Parse each line's timestamp; drop events outside the range. At the projected 60MB log ceiling this is ~500ms on SSD â€” acceptable for a user-invoked CLI. Reverse-read is not implemented in v0.7; if profiling shows it matters, add in v0.8.
2. Regex-parse each remaining line into a structured record. Malformed lines: drop, increment `parseErrors`.
3. Group records by `session_id`. Discard `session=-` records from the correlated stream; keep them under a separate `unscoped` counter.
4. Within each session, walk events chronologically. For every `pre_tool` record with `action` âˆˆ {`deny`, `ask`}:
   - Look forward in the same session within `window_seconds`.
   - **Bystander tools are ignored.** A `post_tool` record whose `tool` is neither `Bash` nor the ctx MCP pattern (e.g., native `Read`, `Grep`, `TodoWrite`) does **not** close the pre_tool and does **not** invalidate it. Keep scanning forward within the window.
   - First in-window `post_tool` with `tool == "Bash"` and `exit == 0` â†’ `bypassed` (deny) / `user_approved` (ask).
   - First in-window `post_tool` with `tool == "Bash"` and `exit != 0` â†’ `bypass_failed` (deny) / `approved_failed` (ask). Reported but excluded from the primary obey-rate denominator.
   - First in-window `post_tool` with `tool` matching the ctx MCP pattern â†’ `obeyed` (deny) / `redirected` (ask).
   - No relevant `post_tool` found in window â†’ `abandoned` (deny) / `canceled` (ask).
5. Each relevant `post_tool` event (Bash or ctx MCP) closes at most one preceding `pre_tool` â€” the earliest unclosed one in the same session. Bystander events never close anything.
6. Aggregate per rule pattern: for each distinct `pattern` string in `pre_tool` events, compute triggers / bypasses / bypass rate.
7. Cache stats: aggregate `cache-write`, `cache-read`, `cache-gc` events in the same range. Hit rate = `cache-read result=hit` / total `cache-read`.

### MCP tool name â€” assumption to verify in plan phase

The ctx MCP pattern above is written as `^mcp__<server>__ctx_(grep|read|shell)$` because the exact `tool_name` string Claude Code emits for plugin-installed MCP tools has not been verified from a live `PostToolUse` payload. The repo is installed as a Claude Code plugin (`claude-code-ctx`), and plugin namespacing may differ from a user-installed MCP server.

### Cache GC (`mcp_cache.js`)

Config block added to `config.default.json`:

```json
"cache": {
  "gc": {
    "enabled": true,
    "ttl_hours": 24,
    "max_bytes": 104857600,
    "sweep_probability": 0.05
  }
}
```

Behavior:

- `writeCache(content, opts, deps)`: with probability `sweep_probability`, call `sweep()` before writing. Then write. Append `cache-write` event to `hooks.log`. `deps.random` defaults to `Math.random`, injectable for tests.
- `readCache(ref)`: read, never sweep. Append `cache-read result=hit|miss bytes=<n>` event to `hooks.log`.
- `sweep()`:
  1. List all entries in the cache dir.
  2. Evict any with `mtime > ttl_hours ago`.
  3. If total size still over `max_bytes`, sort remaining by `mtime` ascending and evict oldest until under threshold.
  4. Append `cache-gc swept=<N> bytes_freed=<M>` event to `hooks.log`.
- Concurrent safety: a single MCP process is serial per handler. Cross-process: open-fd-outlives-unlink on macOS/Linux means a concurrent read completes even if a different process evicts the file.
- `cache.gc.enabled: false` short-circuits `sweep()` to a no-op (read/write events still logged).

Counters are not persisted in a separate file; every metric derives from `hooks.log` events filtered by the report's date range.

### Bash rule expansion

Appended to `hooks.pre_tool_use.rules` in `config.default.json`. All `ask` mode (no per-rule `mode` field, inherits `default_mode: ask`). Patterns:

```json
{ "tool": "Bash", "match": "^\\s*rg\\s+.*\\s(-r|--recursive)(\\s|$)", "reason": "rg recursive is the same footgun as grep -r. Use ctx_grep({pattern, path}) â€” inline if small, summary+ref if big." },
{ "tool": "Bash", "match": "^\\s*grep\\s+-R(\\s|$)",                   "reason": "grep -R (capital) is recursive. Use ctx_grep." },
{ "tool": "Bash", "match": "^\\s*egrep\\s+-r(\\s|$)",                  "reason": "egrep -r same as grep -r. Use ctx_grep." },
{ "tool": "Bash", "match": "^\\s*(awk|sed)\\s+['\"][^'\"]+['\"]\\s+[^-|]", "reason": "awk/sed over a file without a bounded pipeline can emit the full file. Pipe through head -N, or use ctx_read." },
{ "tool": "Bash", "match": "^\\s*wc\\s+-l\\s+[^|]*\\*",                "reason": "wc -l on a glob can stat thousands. Scope to a subdir, or use ctx_shell for a summary." },
{ "tool": "Bash", "match": "^(?!.*-maxdepth)\\s*find\\s+\\.\\s.*-print", "reason": "find without -maxdepth prints everything. Add -maxdepth N or use ctx_shell." }
```

Known coverage gaps and false positives:

- `grep -r foo src | head -10` matches the existing `^\\s*grep\\s+-r` rule even though `head` bounds output via SIGPIPE. One extra `ask` prompt â€” the metric shows if it's a real problem via the `user_approved` rate on that pattern.
- The awk/sed pattern requires the script to be the first argument. Variants like `awk -F, '{print}' file.txt` or `sed -i 's/a/b/' file.txt` (any leading flag) are NOT matched. Accepted gap â€” the Phase 2 data will show if these dominate; if so, v0.8 tightens the pattern.
- The `wc -l` pattern only catches glob-style arguments (`src/**/*.js`). `wc -l $(find . -name '*.js')` and other bounded-but-ugly forms are not matched.

The common theme: v0.7 catches the common-case footguns. Phase 2 measurement is how we learn whether the misses matter.

## Error handling

| Module | Policy | Rationale |
|---|---|---|
| `hooks.js` post_tool logging | `try/catch`, exit 0 | CLAUDE.md invariant: hooks degrade silently |
| `mcp_cache.js` GC sweep | `try/catch`, sweep failure does not abort write | GC is auxiliary; write must succeed |
| `metrics.js` | Loud fail, exit 1 via CLI | User-invoked; silent failure hides bugs |
| `cli.js ctx metrics` | Forwards metrics errors | Standard CLI dispatch |

Edge cases and expected behavior:

- **Log missing/empty** â†’ `ctx metrics` prints `no events recorded yet`. No division by zero on percentages.
- **Log very large** (practical ceiling 60MB at heavy usage; ~10K events/day Ã— 200B Ã— 30d): forward-scan reads the whole file (~500ms SSD at ceiling). v0.7 assumes no rotation and no reverse-read optimization.
- **session_id missing** â†’ log entry writes `session=-`; parser counts these under `unscoped`, excludes from correlation.
- **pre_tool with no matching post_tool in window** â†’ `abandoned` (deny) / `canceled` (ask).
- **Multiple post_tool events in window** â†’ only the earliest closes the preceding pre_tool.
- **Malformed log line** â†’ skipped, `parseErrors++`, trailing note in report.
- **Cache dir missing** â†’ `writeCache` mkdir's; `readCache` returns not-found; `sweep` no-ops on empty dir.
- **Cache read while sweep deletes the file** â†’ open-fd semantics make this safe on Linux/macOS. Documented, not guarded.
- **`pre_tool_use.enabled: false`** â†’ report prints `pre_tool_use disabled in config â€” no data to analyze`.
- **Bash with `exit != 0`** â†’ `bypass_failed` / `approved_failed` buckets; excluded from primary denominators.
- **MCP tool name format** â€” the match regex `/^mcp__ctx__ctx_(grep|read|shell)$/` pins the current Claude Code naming. Test fixture uses the exact string Claude Code emits; if the namespace ever changes, the fixture and regex update together.
- **Unknown hook input shape** (future Claude Code version changes the hook payload) â€” accessors use optional chaining and default values; a missing field degrades to `-` / `0` rather than throwing.

### Backwards compatibility

- `hooks.log` format extension (new `session=` field on pre_tool, new post_tool event type) is additive. The only consumer is `metrics.js`, whose parser treats missing `session=` as `-` and ignores event types it does not recognize.
- User config deep-merges with defaults, but arrays replace (CLAUDE.md invariant). Users who have a custom `pre_tool_use.rules` array will NOT receive the 6 new patterns automatically. Release notes call this out with the exact copy-paste path. No automated migration in v0.7.

### Feature flags

None. Every new behavior has a safe no-op fallback:

- `ctx metrics` on an empty log prints a message, does not error.
- Cache GC can be disabled via `cache.gc.enabled: false`.
- New Bash rules default to `ask` mode â€” user can remove them from config or override `default_mode`.
- post_tool logging adds one `fs.appendFileSync` per tool call; pre_tool already does the same at a comparable rate.
